# 文档维护规范

> 状态：Required  
> 生效日期：2026-07-16

## 目标

文档必须描述当前真实行为，并明确区分已实现、部分完成和规划中。文档维护不是发布后的补写工作，而是行为变更的一部分。

## 变更触发器

出现以下任一变化时，相关文档必须在同一提交更新：

| 变化 | 至少更新 |
|---|---|
| CLI、环境变量、安装命令 | `user-guide.md`、`README.md` |
| HTTP/MCP 参数或返回行为 | `user-guide.md`、`mcp-client-guide.md`、`technical-architecture.md` |
| 领域状态、scope 或权限规则 | `spec.md`、`technical-architecture.md` |
| 数据表、事件、迁移或重放逻辑 | `technical-architecture.md`、`status.md` |
| UI 工作流 | `user-guide.md`、`status.md` |
| 里程碑完成或范围调整 | `status.md`、`spec.md` |
| 新风险、限制或弃用 | 相关指南的限制或故障排查章节 |

## 周期维护

在活跃开发期间执行以下节奏：

- 每次行为变更：提交前检查受影响文档。
- 每周一次：核对 `status.md`、命令、测试数量、已知限制和下一步。
- 每个里程碑：核对 `spec.md` 验收标准并记录未满足项。
- 每次发布：从全新目录按 `user-guide.md` 和 `mcp-client-guide.md` 完成一次冷启动验证。

“每周”是项目维护约定，不代表仓库会自动执行定时任务。需要提醒自动化时，应在项目管理工具中单独创建。

## 文档头信息

长期文档顶部应包含文档状态、最近核对日期，以及适用版本或代码基线（适用时）。状态使用 Draft、Active、Frozen baseline 或 Deprecated。

## 提交前清单

- [ ] 文档中的命令可以从仓库根目录执行。
- [ ] 路径、环境变量、端口和工具名与代码一致。
- [ ] 没有把目标架构写成当前实现。
- [ ] 新参数包含默认值、权限和失败方式。
- [ ] 安全限制没有被弱化或省略。
- [ ] README 和 `docs/README.md` 链接有效。
- [ ] `status.md` 的测试数量和阶段描述已更新。
- [ ] 规格变化有明确理由，而不是为了迁就缺陷。

## 维护责任

实现行为变更的人或 Agent 同时负责首轮文档更新。代码审阅应把“文档与实际行为一致”作为独立检查轴；发现漂移时，优先在同一变更中修复。
